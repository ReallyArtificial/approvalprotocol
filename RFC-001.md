# RFC-001: Approval Protocol Design Philosophy & Future Directions

**Status:** Open for Comments  
**Authors:** Friday (@fridayjoshi), Harsh (@josharsh)  
**Date:** 2026-06-05  
**Version:** 0.1.1  

---

## Abstract

This RFC documents the design philosophy behind Approval Protocol v0.1 and opens discussion on future directions. We've shipped a working implementation (PROTOCOL.md, reference TypeScript server), but several architectural questions remain open:

1. Should rollback be agent-side or server-side?
2. How should multi-approver workflows be modeled?
3. What's the right balance between protocol simplicity and production features?
4. What transports should v0.2 prioritize (WebSocket, stdio, gRPC)?
5. How do we handle escalation chains and conditional routing?

This RFC invites feedback from implementers, adopters, and the broader AI infrastructure community.

---

## 1. Problem Statement

AI agents take actions in the real world — send emails, spend money, modify databases, deploy code. Every agent framework has its own half-baked human-in-the-loop mechanism:

- **LangChain** — `HumanInputRun` tool, synchronous CLI prompts only
- **CrewAI** — `human_input=True` flag, no policy layer
- **OpenAI Assistants API** — `requires_action` state, polling-only
- **Custom frameworks** — bespoke approval flows, tightly coupled to the app

Problems:
- **Framework lock-in** — approval logic is tied to the agent framework
- **No audit trail** — decisions aren't logged, making compliance impossible
- **Missing channels** — most support only CLI or webhook, no Slack/email/SMS
- **Incomplete lifecycle** — no standardized confirmation or rollback
- **No execution modes** — can't specify sandbox/dry-run without custom code

**Approval Protocol solves this by being a protocol, not a library.** It defines message formats and flows that work with any framework, any language, any human interface.

---

## 2. Design Philosophy

### 2.1 Protocol over Library

AP is a **protocol** (like HTTP or MCP), not a library (like `prompt-toolkit` or `inquirer`). This means:

- **Language-agnostic** — Reference implementation is TypeScript, but Python/Go/Rust implementations should be straightforward
- **Transport-agnostic** — v0.1 uses HTTP + polling, but WebSocket, stdio, gRPC are all valid transports
- **Framework-agnostic** — Works with LangChain, OpenAI SDK, CrewAI, or raw HTTP

**Why this matters:**  
If AP were a library, every agent framework would need framework-specific integrations. As a protocol, any HTTP client can speak it.

### 2.2 Full Lifecycle, Not Just Approval

Most HITL systems stop at approval. AP covers the **entire action lifecycle**:

```
negotiate → request → decide → execute → confirm → [rollback?]
```

- **`confirm`** closes the loop — agents report back what happened
- **`rollback`** enables reversal — undo metadata captured at confirmation time
- **Execution modes** (`live`, `sandbox`, `dry_run`) let policies specify how actions run

**Why this matters:**  
Without confirmation, you can't audit what actually happened (agent might fail silently).  
Without rollback, you can't recover from mistakes (email sent to wrong recipient).  
Without execution modes, you can't test dangerous actions safely (delete in sandbox first).

### 2.3 Simple First, Production Later

v0.1 deliberately omits features that production deployments need:

- No authentication (run behind an auth proxy)
- No WebSocket transport (polling only)
- No expression parser (uses `new Function()` for conditionals)
- No multi-approver workflows
- No escalation chains

**Why?**  
Getting the core protocol right matters more than shipping all features. Once the message format is stable, adding WebSocket or multi-approver is backwards-compatible.

**Trade-off:**  
Some adopters will need to fork or wrap the reference implementation. That's okay — the protocol is the product, not the implementation.

## 3. Open Questions

### 3.1 Rollback: Agent-Side or Server-Side?

**Current behavior (v0.1):**  
The AP server marks the request as `rolled_back` and returns undo instructions. The **caller** (agent or human operator) is responsible for executing the reversal.

**Why?**  
- Agents know how to undo their own actions better than the server does
- Server doesn't need API credentials for every external service
- Keeps the server stateless-ish (no need to track rollback execution)

**Open question:**  
Should the server optionally execute rollbacks itself?

**Use case:**  
Admin wants to roll back 50 actions at once. Asking 50 agents to individually execute undo is fragile.

**Possible v0.2 design:**
- Add optional `ap/execute-rollback` method
- Server calls agent's webhook or invokes undo API directly
- Requires server to have credentials or agent webhook URLs

**Feedback requested:**  
Which model do you prefer? Agent-side only, server-side optional, or both?

---

### 3.2 Multi-Approver Workflows

**Use case:**  
"Database schema changes require approval from both a DBA and a security engineer."

**Current behavior (v0.1):**  
Not supported. One request → one human → one decision.

**Possible v0.2 designs:**

**Option A: Serial approvers**
```json
{
  "approvers": [
    { "required": "dba@company.com" },
    { "required": "security@company.com" }
  ],
  "mode": "all"
}
```
Each approver gets notified in sequence. Status transitions: `pending_dba` → `pending_security` → `approved`.

**Option B: Parallel approvers with quorum**
```json
{
  "approvers": ["alice", "bob", "carol"],
  "quorum": 2
}
```
All three get notified simultaneously. First 2 approvals unlock the action.

**Option C: Escalation chain**
```json
{
  "approvers": [
    { "tier": "engineer", "timeout": 300 },
    { "tier": "manager", "timeout": 600 },
    { "tier": "director", "timeout": null }
  ]
}
```
If engineer doesn't respond in 5 minutes, escalate to manager, then director.

**Trade-offs:**
- **Serial** — simpler to implement, but slow
- **Parallel** — faster, but requires more complex state management
- **Escalation** — realistic, but requires timeout handling + notification retry

**Feedback requested:**  
Which model(s) matter most for your use case? Should v0.2 support all three?

### 3.3 Transports: WebSocket, stdio, or Both?

**Current (v0.1):** HTTP + polling  
**Problem:** Agents poll `/ap/status` every N seconds, wasting API calls.

**Option A: WebSocket**
- Agent opens WebSocket connection after `ap/request`
- Server pushes decision when human responds
- Eliminates polling, real-time notifications

**Option B: stdio (like MCP)**
- Server runs as a subprocess
- Agent sends/receives via stdin/stdout JSON-RPC
- Useful for local agents, no network involved

**Option C: HTTP callbacks (webhook-style)**
- Agent provides `callback_url` in `ap/request`
- Server POSTs decision to that URL when ready
- Requires agent to expose HTTP endpoint

**Trade-offs:**
- **WebSocket** — best for latency, adds connection management complexity
- **stdio** — best for local agents, doesn't work for distributed systems
- **HTTP callbacks** — simplest for server, requires agent to be reachable

**Feedback requested:**  
Which transport(s) should v0.2 prioritize? Are there others (gRPC, SSE, NATS)?

### 3.4 Conditional Policies: Expression Language or Code?

**Current (v0.1):**  
Conditional policies use JavaScript expressions evaluated via `new Function()`:

```json
{
  "send_email": {
    "requires": "conditional",
    "when": "params.to.endsWith('@company.com') ? false : true"
  }
}
```

**Problem:**  
`new Function()` is unsafe in production. Need a proper expression parser.

**Options:**

**Option A: Simple expression language (CEL, JSONLogic)**
- Use Common Expression Language (Google's CEL) or JSONLogic
- Sandboxed, no arbitrary code execution
- Example: `params.to.endsWith("@company.com") == false`

**Option B: Policy as code (Python/JS functions)**
- Policies are actual code files
- Server evals them in a sandbox (Deno, QuickJS, gVisor)
- More flexible but harder to secure

**Trade-off:**  
Expression languages are safer but less expressive. Policy-as-code is powerful but risky.

**Feedback requested:**  
Which approach do you prefer? Should v0.2 support both?

---

### 3.5 Undo Metadata: How Much Should the Protocol Define?

**Current (v0.1):**  
Undo metadata is freeform JSON:

```json
{
  "type": "api_call",
  "description": "Recall the email via Gmail API",
  "instructions": {
    "method": "POST",
    "url": "https://gmail.googleapis.com/recall/..."
  }
}
```

**Problem:**  
`instructions` format is unspecified. Every agent invents its own structure.

**Options:**

**Option A: Keep it freeform**
- Flexibility wins, interoperability loses
- Agents understand their own undo format, server doesn't care

**Option B: Standardize undo types**
- Define schemas for common patterns: `http_request`, `sql_query`, `shell_command`, `manual`
- Example:
  ```json
  {
    "type": "http_request",
    "http": {
      "method": "POST",
      "url": "https://api.example.com/undo",
      "headers": { "Authorization": "Bearer ..." },
      "body": { "action_id": "123" }
    }
  }
  ```

**Trade-off:**  
Standardization enables server-side rollback execution. Flexibility enables agent-specific undo logic.

**Feedback requested:**  
Should the protocol define standard undo schemas, or leave it agent-specific?

## 4. Security & Production Hardening

v0.1 is **not production-ready** by design. Here's what v0.2+ needs:

### 4.1 Authentication
- **API keys** — Basic auth for agent → server requests
- **HMAC-signed webhooks** — Verify channel callbacks aren't spoofed
- **Human identity** — Tie decisions to real users (OIDC, SAML)

### 4.2 Conditional Expression Safety
- Replace `new Function()` with CEL or JSONLogic
- Sandbox untrusted expressions (no I/O, no network, no file access)

### 4.3 Audit Compliance
- Immutable audit log (append-only)
- Export to external systems (Splunk, S3, BigQuery)
- Retention policies (GDPR, SOC2)

### 4.4 Deployment Patterns
- **Centralized** — One AP server for many agents (like SSO)
- **Embedded** — AP server runs in-process with the agent
- **Multi-tenant** — One server, many teams/orgs (needs auth + isolation)

**Feedback requested:**  
Which deployment pattern matters most for your use case?

## 5. Roadmap

### v0.2 (Next, ~Q3 2026)
- WebSocket transport (real-time decisions, no polling)
- stdio transport (MCP-style, for local agents)
- Proper expression parser (CEL or JSONLogic)
- Escalation chains (timeout → escalate to higher tier)
- Multi-approver workflows (serial, parallel, quorum)

### v0.3 (Future)
- gRPC transport (for high-throughput systems)
- Server-side rollback execution (optional)
- Standard undo schemas (http_request, sql_query, shell_command)
- Slack native channel (no webhook glue code)
- Web dashboard (view audit log, approve from browser)

### v1.0 (Long-term)
- Authentication & authorization (API keys, OIDC, RBAC)
- Multi-tenant support (one server, many orgs)
- Immutable audit log with cryptographic proofs
- Production reference implementations (Go, Python, Rust)

**Not on roadmap:**
- AI-assisted approval (auto-approve based on ML model)
- Blockchain-based audit trail (unnecessary complexity)
- GraphQL API (HTTP JSON-RPC is enough)

## 7. Prior Art & References

**Related protocols:**
- **MCP (Model Context Protocol)** — Anthropic's protocol for tool access
- **OAuth 2.0 Device Flow** — User approval for headless devices
- **GitHub Actions approval environments** — Deployment protection rules
- **PagerDuty escalation chains** — Multi-tier notification + approval

**Related libraries:**
- **LangChain `HumanInputRun`** — CLI-only, synchronous approval
- **OpenAI Assistants `requires_action`** — Polling-based approval, no rollback
- **AWS Step Functions approval tasks** — Webhook-based approval, no undo

**What AP adds:**
- Full lifecycle (approval + confirmation + rollback)
- Protocol (not library)
- Channel-agnostic (CLI, Slack, email, custom)
- Execution modes (live, sandbox, dry_run)
- Undo metadata (machine-readable reversal instructions)

---

## 8. Conclusion

Approval Protocol v0.1 is a working implementation of a simple, framework-agnostic HITL protocol. It's production-adjacent but not production-ready.

**What we got right:**
- Full action lifecycle (not just approval)
- Protocol over library (works with any framework)
- Simple core (6 methods, JSON over HTTP)

**What's still open:**
- Rollback execution (agent-side or server-side?)
- Multi-approver workflows (serial, parallel, quorum, escalation?)
- Transports (WebSocket, stdio, gRPC?)
- Conditional policies (expression language or code?)
- Undo metadata (freeform or standardized schemas?)

**We need your input.** Comment on this RFC. Build something with AP. Tell us what breaks. Tell us what's missing. Help us make this the standard HITL protocol for AI agents.
